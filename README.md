# NestMatch API — v1

Buyer-side real estate matching engine for Sydney.

## Structure

```
nestmatch/
├── app/
│   ├── main.py          # FastAPI app + /search endpoint
│   ├── engine.py        # Matching engine (filters → scores → explanations)
│   ├── models.py        # Pydantic request / response models
│   └── seed_data.py     # 10 Sydney seed properties
├── tests/
│   └── test_engine.py   # Unit + integration tests
├── schema.sql           # PostgreSQL schema + seed INSERT statements
├── requirements.txt
└── README.md
```

## Quick start

```bash
# 1. Create virtual environment
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Run the API
uvicorn app.main:app --reload

# API now live at http://localhost:8000
# Interactive docs at http://localhost:8000/docs
```

## Run tests

```bash
pytest tests/ -v
```

## Example request

```bash
curl -X POST http://localhost:8000/search \
  -H "Content-Type: application/json" \
  -d '{
    "budget": {
      "min_price": 900000,
      "max_price": 1400000
    },
    "property": {
      "property_type": "apartment",
      "min_bedrooms": 2,
      "min_internal_size_sqm": 95,
      "new_build": false

    },
    "lifestyle": {
      "school_priority": true,
      "noise_preference": "quiet",
      "parking_required": true
    }
  }'
```

## Matching logic

### Stage 1 — Hard filters
Candidates are eliminated if they:
- Exceed the buyer's maximum budget
- Are the wrong property type
- Have fewer bedrooms than required
- Are more than 10 sqm below the size minimum (tolerance band)
- Have no parking when parking is required
- Are a new build when buyer wants established (or vice versa)

### Stage 2 — Feature scores (0–1 each)

| Score | Weight | Logic |
|---|---|---|
| price_fit | 30% | Distance from budget midpoint; 0 if over max |
| commute_score | 20% | Linear decay from 0.5× to 1.5× the commute limit |
| size_score | 15% | Perfect at 1.2× minimum; penalised below |
| transport_score | 15% | Property score × buyer preference multiplier |
| school_score | 10% | Pass-through; neutral 0.5 if not a priority |
| lifestyle_fit | 10% | Noise alignment + family score blend |

### Stage 3 — Explanation
All highlights and trade-offs are generated by deterministic rules — no LLM,
no randomness. The same input always produces the same output.

## Database

Run `schema.sql` against any Postgres 15+ instance (Neon, Supabase, or local).
The seed data in the SQL file mirrors `app/seed_data.py` exactly.

The v1 API reads from the in-memory seed list. To wire up Postgres, replace
the `SEED_PROPERTIES` import in `main.py` with a DB query using asyncpg or
SQLAlchemy.
